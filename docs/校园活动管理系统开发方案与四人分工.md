# 校园活动管理系统开发方案与四人分工

## 1. 项目概述

本项目为“校园活动管理系统”，主要面向高校社团、学院部门和校级组织的校园活动管理场景。系统需要完成从活动发布、学生报名、现场签到、活动评分到数据统计与活动推荐的完整流程，解决传统活动管理中信息分散、报名统计低效、签到核验困难、活动质量难以评估等问题。

系统主要使用对象包括学生和组织者。学生可以登录系统、浏览活动、查看活动详情、报名活动、取消报名、参与签到、提交评分反馈和查看推荐活动；组织者可以创建活动、维护活动信息、查看报名名单、管理签到、查看签到率、平均评分和评价反馈。管理员角色可作为扩展功能，用于账号状态管理、违规活动审核和系统统计报表管理。

系统核心业务闭环如下：

```text
组织者创建活动
        ↓
组织者发布活动
        ↓
学生浏览活动并报名
        ↓
系统生成报名记录
        ↓
活动现场签到
        ↓
系统生成签到记录
        ↓
活动结束后评分反馈
        ↓
系统生成统计结果和推荐数据
```

## 2. 技术栈选择

本系统建议采用 B/S 架构，即浏览器访问系统，后端提供接口服务，数据库保存业务数据。这样更适合多人分工开发，也方便最后进行演示。

### 2.1 前端技术栈

| 技术 | 作用 |
|---|---|
| Vue 3 | 构建前端页面和组件 |
| Vite | 前端项目构建工具，启动速度快 |
| Element Plus | 快速搭建表单、表格、弹窗、分页等后台管理界面 |
| Axios | 调用后端接口 |
| Vue Router | 实现登录页、活动列表页、详情页、管理页等页面跳转 |
| Pinia / localStorage | 保存登录用户信息、角色信息和 Token |

前端主要负责学生端和组织者端页面展示，包括登录页面、活动列表页面、活动详情页面、报名记录页面、活动管理页面、报名名单页面、签到管理页面、评分反馈页面和统计页面。

### 2.2 后端技术栈

| 技术 | 作用 |
|---|---|
| Java 17 | 后端主要开发语言 |
| Spring Boot 3 | 快速搭建 Web 后端项目 |
| Spring Web | 编写 RESTful API 接口 |
| MyBatis-Plus | 简化数据库增删改查 |
| Maven | 项目依赖管理和打包 |
| Lombok | 简化实体类、DTO、VO 的代码 |
| Spring Validation | 参数校验，例如时间、人数上限、评分范围等 |

后端主要负责业务逻辑处理，包括登录验证、活动状态判断、报名校验、签到校验、评分限制、统计计算和推荐规则实现。

### 2.3 数据库技术栈

| 技术 | 作用 |
|---|---|
| SQL Server | 保存系统核心业务数据 |
| SSMS | 数据库管理、建表、执行 SQL 脚本 |
| JDBC Driver | 后端连接 SQL Server 数据库 |
| 数据库约束 | 保证主键、外键、唯一性和数据一致性 |

数据库主要设计学生表、组织者表、活动表、报名记录表、签到记录表和评分记录表。为了防止重复报名、重复签到和重复评分，需要设置联合唯一约束或唯一约束。

如果后续老师允许使用 MySQL，也可以将数据库替换为 MySQL 8.0，整体后端代码改动不大。

## 3. 系统功能模块划分

根据需求分析和概念逻辑结构设计，系统可以划分为以下几个模块。

| 模块 | 主要功能 |
|---|---|
| 用户管理模块 | 学生和组织者登录、身份区分、基础信息维护、账号状态控制 |
| 活动管理模块 | 活动创建、发布、修改、取消、查询、详情展示 |
| 报名管理模块 | 学生报名、取消报名、报名校验、报名名单查看 |
| 签到管理模块 | 签到码生成、学生签到、人工补签、签到记录维护 |
| 评分反馈模块 | 已签到学生评分、文字评价、评分记录维护 |
| 活动推荐模块 | 按类别、热度、评分和历史参与记录进行简单推荐 |
| 统计查询模块 | 报名人数、签到人数、签到率、平均评分、评价明细统计 |

其中，用户管理、活动管理、报名管理和签到管理属于系统必须完成的核心功能；评分反馈、统计查询和活动推荐可以作为增强功能，但最好至少实现一个简单版本，方便最后展示系统完整性。

## 4. 数据库表设计概述

系统核心数据表如下：

| 表名 | 中文含义 | 主要字段 |
|---|---|---|
| student | 学生表 | student_id、student_no、student_name、college、major、grade、phone、email、account_status、created_time |
| organizer | 组织者表 | organizer_id、organizer_name、organizer_type、contact_name、contact_phone、account_status、created_time |
| activity | 活动表 | activity_id、activity_title、category、description、organizer_id、location、start_time、end_time、register_start_time、register_end_time、capacity、activity_status |
| registration | 报名记录表 | registration_id、activity_id、student_id、registration_time、registration_status、cancel_time |
| checkin | 签到记录表 | checkin_id、registration_id、activity_id、student_id、checkin_time、checkin_method、checkin_status、remark |
| rating | 评分记录表 | rating_id、activity_id、student_id、checkin_id、score、comment、rating_time |

主要表关系如下：

```text
organizer 1 —— N activity
student   1 —— N registration
activity  1 —— N registration
registration 1 —— 0/1 checkin
checkin 1 —— 0/1 rating
student 1 —— N rating
activity 1 —— N rating
```

主要约束建议如下：

| 约束 | 作用 |
|---|---|
| student.student_no 唯一 | 防止学号重复 |
| activity.organizer_id 外键 | 活动必须属于某个组织者 |
| registration.student_id + registration.activity_id 唯一 | 防止同一学生重复报名同一活动 |
| checkin.registration_id 唯一 | 防止同一报名记录重复签到 |
| rating.student_id + rating.activity_id 唯一 | 防止同一学生对同一活动重复评分 |
| rating.score 范围 1 到 5 | 限制评分合法性 |

## 5. 整个系统开发流程

### 5.1 第一阶段：需求确认与原型设计

本阶段主要完成系统功能边界确认和页面规划。小组需要先明确哪些功能必须做，哪些功能作为扩展功能。建议必须完成学生登录、活动浏览、活动报名、组织者发布活动、报名名单查看、签到、评分和统计功能；推荐模块可以先采用简单规则实现。

本阶段产出内容包括系统模块划分、用户角色说明、页面清单、核心业务流程图和接口初步清单。

### 5.2 第二阶段：数据库设计与初始化

本阶段根据概念结构设计和逻辑结构设计完成数据库建表。先建立学生表、组织者表、活动表、报名记录表、签到记录表和评分记录表，再设置主键、外键和唯一约束，最后插入测试数据。

数据库完成后，需要准备一些测试数据，例如 5 个学生账号、2 个组织者账号、5 个活动、若干报名记录、签到记录和评分记录。这样后续前后端开发时可以直接进行测试。

### 5.3 第三阶段：后端接口开发

后端开发按照“基础接口先行、业务接口后接”的顺序进行。先完成登录、用户信息查询、活动列表和活动详情接口，再完成报名、取消报名、签到、评分和统计接口。

建议后端接口统一采用 RESTful 风格，例如：

```text
POST   /api/login
GET    /api/activities
GET    /api/activities/{id}
POST   /api/activities
PUT    /api/activities/{id}
POST   /api/registrations
PUT    /api/registrations/{id}/cancel
POST   /api/checkins
POST   /api/ratings
GET    /api/statistics/activity/{id}
GET    /api/recommendations
```

### 5.4 第四阶段：前端页面开发

前端页面可以分为学生端和组织者端。学生端主要包括登录页、活动列表页、活动详情页、我的报名页、签到页、评分页和推荐活动页；组织者端主要包括活动管理页、活动发布页、报名名单页、签到管理页和活动统计页。

前端开发时先使用静态假数据搭建页面，等后端接口完成后再替换为真实接口数据。这样可以减少前后端互相等待的时间。

### 5.5 第五阶段：前后端联调

联调时按照业务流程逐步测试，不要一开始就测试全部功能。建议先测试登录，再测试活动发布和活动查询，然后测试报名和取消报名，接着测试签到和人工补签，最后测试评分、统计和推荐。

联调重点检查以下问题：

| 检查项 | 说明 |
|---|---|
| 登录身份是否正确 | 学生和组织者进入不同页面 |
| 活动状态是否正确 | 草稿、报名中、已结束、已取消等状态显示正确 |
| 报名校验是否完整 | 重复报名、名额不足、报名时间过期、时间冲突都要拦截 |
| 签到校验是否完整 | 未报名不能签到，重复签到不能成功 |
| 评分校验是否完整 | 未签到不能评分，重复评分不能提交 |
| 统计数据是否正确 | 报名人数、签到人数、签到率、平均分能正确计算 |

### 5.6 第六阶段：测试、修改与文档整理

最后阶段需要完成系统测试、截图整理、演示流程准备和课程报告编写。测试时建议每个成员至少负责自己模块的功能测试，同时由一名成员按照完整业务流程进行总测试。

最终需要准备的材料包括：

| 材料 | 说明 |
|---|---|
| 系统需求分析文档 | 已完成，可适当补充技术选型 |
| 概念与逻辑结构设计文档 | 已完成，可补充表关系图和约束说明 |
| SQL 建表脚本 | 包含建表、主外键、唯一约束和测试数据 |
| 系统源代码 | 前端代码、后端代码、配置文件 |
| 接口文档 | 说明每个接口的路径、参数和返回值 |
| 系统测试截图 | 登录、活动发布、报名、签到、评分、统计等截图 |
| 最终答辩 PPT | 展示系统背景、功能模块、数据库设计、开发实现和运行效果 |

## 6. 四人分工方案

为了让 4 个人工作量相对均衡，建议按照“数据库底座 + 活动管理 + 报名流程 + 签到评分统计”进行分工，而不是简单地分成前端和后端。这样每个人都有独立模块，也能减少互相等待。

| 成员 | 负责方向 | 主要任务 | 交付成果 |
|---|---|---|---|
| 成员 A | 数据库与用户权限 | 负责数据库建表、主外键约束、测试数据、学生和组织者登录、角色权限控制 | SQL 脚本、测试数据、登录接口、用户信息接口 |
| 成员 B | 活动管理模块 | 负责组织者创建活动、修改活动、发布活动、取消活动；负责学生查看活动列表和活动详情 | 活动管理接口、活动发布页面、活动列表页面、活动详情页面 |
| 成员 C | 报名管理模块 | 负责学生报名、取消报名、报名时间校验、名额校验、重复报名校验、时间冲突校验、报名名单查询 | 报名接口、取消报名接口、报名名单接口、我的报名页面 |
| 成员 D | 签到、评分、推荐与统计 | 负责签到码签到、人工补签、签到记录、评分反馈、活动统计和简单推荐 | 签到接口、评分接口、统计接口、推荐接口、签到和统计页面 |

## 7. 各成员详细任务说明

### 7.1 成员 A：数据库与用户权限

成员 A 是系统底座负责人，需要先完成数据库设计和后端基础结构。其他成员的功能都依赖数据库表和基础接口，因此成员 A 应该最先开始。

主要任务包括：

```text
1. 建立 student、organizer、activity、registration、checkin、rating 六张核心表。
2. 设置主键、外键、唯一约束和必要的检查约束。
3. 插入测试数据，保证每个模块都能测试。
4. 完成学生和组织者登录功能。
5. 根据用户角色控制页面和接口访问权限。
6. 统一后端返回格式，例如 code、message、data。
```

建议成员 A 还要负责维护接口文档和数据库文档，因为他最清楚表结构和字段含义。

### 7.2 成员 B：活动管理模块

成员 B 负责活动从创建到发布的全过程，是组织者端的核心功能负责人。

主要任务包括：

```text
1. 组织者创建活动。
2. 组织者修改自己发布的活动。
3. 组织者发布或取消活动。
4. 学生查看活动列表。
5. 学生按照类别、时间、地点、组织者等条件查询活动。
6. 学生查看活动详情，包括活动时间、地点、剩余名额和组织者信息。
```

该成员需要和成员 C 对接，因为报名模块需要读取活动状态、报名开始时间、报名截止时间、人数上限和剩余名额。

### 7.3 成员 C：报名管理模块

成员 C 负责学生报名和退报，是系统业务逻辑最集中的部分之一。

主要任务包括：

```text
1. 学生对活动进行报名。
2. 学生在报名截止前取消报名。
3. 校验学生账号状态是否正常。
4. 校验活动是否处于可报名状态。
5. 校验当前时间是否在报名时间范围内。
6. 校验活动剩余名额是否大于 0。
7. 校验学生是否已经报名过该活动。
8. 校验学生报名的活动时间是否存在冲突。
9. 组织者查看报名名单，并按照报名状态筛选。
```

成员 C 的功能需要和成员 B 的活动状态、活动时间、人数上限进行联动，也需要为成员 D 的签到模块提供有效报名记录。

### 7.4 成员 D：签到、评分、推荐与统计模块

成员 D 负责活动报名之后的流程，包括签到、评分和统计展示。虽然功能较多，但可以先完成基础版，保证系统完整闭环。

主要任务包括：

```text
1. 组织者生成签到码。
2. 学生输入签到码完成签到。
3. 系统校验学生是否已报名、报名状态是否有效、是否重复签到。
4. 组织者对特殊情况进行人工补签，并填写备注。
5. 已签到学生在活动结束后提交评分和评价。
6. 系统限制评分范围为 1 到 5 分。
7. 系统限制同一学生对同一活动只能评分一次。
8. 统计报名人数、签到人数、签到率、平均评分和评价明细。
9. 使用简单规则生成推荐活动列表。
```

推荐模块可以先按照以下规则实现：

```text
1. 优先推荐学生历史报名类别相同的活动。
2. 其次推荐报名人数较多的热门活动。
3. 再推荐平均评分较高的活动。
4. 排除已结束、已取消、已报名或不可报名的活动。
```

## 8. 模块之间的接口依赖关系

为了避免后期联调混乱，建议提前明确模块之间的依赖关系。

| 依赖方 | 被依赖方 | 依赖内容 |
|---|---|---|
| 成员 B | 成员 A | 需要用户登录信息和组织者身份 |
| 成员 C | 成员 A | 需要学生身份和账号状态 |
| 成员 C | 成员 B | 需要活动状态、报名时间、人数上限、活动时间 |
| 成员 D | 成员 A | 需要学生和组织者身份 |
| 成员 D | 成员 B | 需要活动状态和活动时间 |
| 成员 D | 成员 C | 需要有效报名记录 |
| 全体成员 | 成员 A | 需要统一数据库字段、接口返回格式和测试数据 |

## 9. 建议开发时间安排

如果开发周期按照 2 周左右安排，可以采用下面的进度。

| 第 13 天 | 整理截图、SQL 脚本、接口文档和项目报告 |
| 第 14 天 | 准备演示流程和答辩 PPT |

## 10. 最终演示流程建议

最终演示时可以按照一条完整业务线展示，不要零散展示页面。

```text
1. 组织者登录系统。
2. 组织者创建并发布一个校园活动。
3. 学生登录系统。
4. 学生浏览活动列表并查看活动详情。
5. 学生报名该活动。
6. 组织者查看报名名单。
7. 活动开始后，组织者生成签到码。
8. 学生输入签到码完成签到。
9. 活动结束后，学生提交评分和评价。
10. 组织者查看报名人数、签到人数、签到率、平均评分和评价内容。
11. 学生查看系统推荐的其他活动。
```

这个演示顺序能够完整体现系统从活动发布到活动结束后的全过程，也能体现数据库中活动表、报名记录表、签到记录表和评分记录表之间的关系。

## 11. 总结

本系统建议采用 Vue 3 + Spring Boot + SQL Server 的前后端分离技术栈进行开发。系统核心功能围绕学生、组织者、活动、报名记录、签到记录和评分记录展开，能够实现校园活动从发布、报名、签到到评分统计的完整管理流程。

四人分工上，建议成员 A 负责数据库和用户权限，成员 B 负责活动管理，成员 C 负责报名管理，成员 D 负责签到、评分、推荐和统计。这样的分工既符合系统模块结构，也能让每个人负责一个相对完整的业务部分，最后通过统一接口和数据库表进行整合，形成一个完整的校园活动管理系统。
